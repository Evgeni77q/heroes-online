# Purpose

The Account domain is responsible for the identity of a human user within Heroes Online.

An Account represents the real user, not a game character.

Every Account may own one or more Players across multiple Worlds (depending on game rules).

The Account domain is the root of authentication, authorization and security.

---

# Responsibilities

The Account module is responsible for:

- account registration;
- authentication;
- password management;
- email verification;
- session management;
- account status;
- account security;
- linked authentication providers;
- acceptance of legal agreements.

The Account module is NOT responsible for gameplay.

Gameplay begins at the Player aggregate.

---

# Aggregate Root

```
Account
```

The Account aggregate is the authoritative source of identity.

All operations affecting account state must pass through this aggregate.

No external module may modify account data directly.

---

# Domain Entity

## Account

### Description

Represents one registered user.

The Account exists independently of worlds, cities or heroes.

Deleting a Player must never delete an Account.

Deleting a World must never delete an Account.

# Invariants

The following business rules must always be true.

## Identity

- Every Account has exactly one AccountId.
- AccountId is immutable.
- AccountId is generated only by the backend.

---

## Email

- Email is required.
- Email is globally unique.
- Email comparison is case-insensitive.
- Email is stored in normalized lowercase form.
- Changing an email invalidates its verification status.

---

## Username

- Username is required.
- Username is globally unique.
- Username comparison is case-insensitive.
- Username cannot be reused while an account exists.
- Username changes are restricted by platform policy.

---

## Password

- Passwords are never persisted.
- Only Argon2id password hashes are stored.
- Password hashes are never returned by any API.
- Password changes invalidate all active sessions except the current one (configurable).

---

## Account Status

Only valid state transitions are allowed.

```
PendingVerification
        │
        ▼
     Active
      │ │
      │ └──────────────► Locked
      │                    │
      │                    ▼
      │                 Active
      │
      ├──────────────► Suspended
      │                    │
      │                    ▼
      └────────────────► Deleted
```

Deleted is terminal.

A deleted account cannot return to Active.

# Domain Behaviors

The Account aggregate exposes the following commands.

## RegisterAccount

Creates a new Account.

Preconditions

- email is unique;
- username is unique;
- password satisfies policy.

Postconditions

- Account created;
- verification email scheduled;
- AccountStatus = PendingVerification.

---

## VerifyEmail

Marks account email as verified.

Preconditions

- verification token valid;
- account exists;
- account not deleted.

Postconditions

- emailVerified = true;
- emailVerifiedAt assigned;
- status changes to Active (if applicable).

---

## ChangePassword

Changes password hash.

Preconditions

- current password valid;
- account active.

Postconditions

- password hash replaced;
- passwordUpdatedAt updated;
- active sessions invalidated according to security policy.

---

## LockAccount

Temporary security lock.

Reasons include:

- brute-force protection;
- suspicious activity;
- administrative action.

---

## UnlockAccount

Returns a locked account to Active when the lock reason is resolved.

---

## SuspendAccount

Administrative operation.

Suspended accounts cannot authenticate.

Gameplay data remains intact.

---

## DeleteAccount

Performs a soft delete.

No gameplay data is removed immediately.

Physical deletion, if ever supported, is handled by a separate administrative process.

# Relationships

The Account aggregate owns or references the following domain relationships.

---

## Player

Relationship

```
Account (1) ------ (N) Player
```

Description

One Account may own multiple Players.

The maximum number of Players is defined by game configuration.

The Account aggregate never stores gameplay state.

Gameplay belongs exclusively to the Player aggregate.

---

## Session

Relationship

```
Account (1) ------ (N) Session
```

Description

An Account may have multiple active sessions depending on security policy.

Sessions are managed by the Authentication module.

The Account aggregate owns the security rules but not the session lifecycle.

---

## Authentication Providers

Relationship

```
Account (1) ------ (N) AuthenticationProvider
```

Examples

- Email + Password
- Google
- Steam
- Discord
- Apple
- Future providers

Authentication providers are optional extensions.

The Account aggregate remains the canonical identity.

---

## Audit Log

Relationship

```
Account (1) ------ (N) AuditRecord
```

Every security-sensitive operation must generate an immutable audit record.

Examples

- Login
- Logout
- Password change
- Email change
- Two-factor enabled
- Two-factor disabled
- Account lock
- Administrative suspension

Audit records are append-only.

They are never modified.

# Domain Policies

## Registration Policy

A registration request is accepted only when:

- email is valid;
- email is unique;
- username is valid;
- username is unique;
- password satisfies password policy;
- required legal agreements are accepted.

---

## Password Policy

Minimum length

12 characters

Required

- uppercase letter
- lowercase letter
- number
- special character

Forbidden

- username inside password
- email inside password
- leaked passwords (future integration)
- previously used passwords (configurable)

---

## Login Policy

Authentication succeeds only if:

- account exists;
- account status allows login;
- email verified (configurable);
- password matches;
- account not locked.

Failed logins increase the failed login counter.

---

## Lockout Policy

Example configuration

```
5 failed attempts

↓

Account locked

↓

15 minutes
```

Values are configurable.

Hardcoded limits are prohibited.

---

## Username Policy

Usernames are immutable by default.

Future paid username changes may be implemented through a dedicated domain service.

Historical usernames must remain reserved for audit purposes.

# Security Model

The Account module is a critical security boundary of the system.

---

## Threat Assumptions

The system assumes the following threats:

- brute-force attacks
- credential stuffing
- session hijacking
- account enumeration attempts
- malicious admin actions
- replay attacks
- API abuse

---

## Protection Mechanisms

### Password Protection

- Argon2id hashing
- configurable memory cost
- configurable iteration cost
- per-environment salt policy

---

### Rate Limiting

All authentication endpoints must implement rate limiting.

Examples:

- login attempts per IP
- login attempts per account
- registration rate limits

---

### Session Security

Sessions must:

- be cryptographically secure
- be revocable
- expire after inactivity
- support rotation

---

### Token Handling

If JWT is used:

- tokens must be short-lived
- refresh tokens must be rotated
- token revocation must be supported

---

### Audit Integrity

Audit logs are:

- append-only
- tamper-resistant
- immutable
- time-stamped

---

## Data Protection

Sensitive fields include:

- password hashes
- session tokens
- email addresses
- IP addresses

These must:

- never be exposed via public API
- never appear in logs
- never be cached in frontend state
