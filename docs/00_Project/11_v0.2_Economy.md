# v0.2 — Economy

**Status:** Planned (after `v0.1.0-alpha`)  
**Prerequisite:** CI `release-gate.yml` green on clean environment

---

# 1. Goal

Make the player **make economic decisions**, not just watch numbers.

Economy must answer:

| Question | Player experience |
|----------|-------------------|
| What is best to upgrade now? | Meaningful trade-offs |
| What should I save resources for? | Goals and planning |
| What limits my growth? | Bottlenecks (storage, production) |
| How does production change over time? | Visible progression |
| Which choices give an advantage? | Strategy, not grind |

This is **game design and balance** — not new architecture.

---

# 2. Infrastructure Rule

**Do not add new infrastructure until a real problem appears.**

For v0.2, do **not** add (unless proven necessary):

- new job queues;
- new Event Bus layers;
- new base services;
- new infrastructural abstractions.

First: prove the existing foundation carries economy growth.

Reuse:

```
Command → Timed Job → Game Loop → Domain Event → WebSocket → UI
GameJobRepository · DomainEventBus · BalanceService · release:gate
```

---

# 3. Likely Scope (stories TBD)

Economy v0.2 is expected to stress:

| Area | Why |
|------|-----|
| `BalanceService` | Central balance node |
| `ResourceProcessor` (Game Loop) | Production ticks |
| Storage limits | Growth bottlenecks |
| WebSocket frequency | `resources.updated` events |
| UI reactivity | Live production without F5 |

Stories will be sliced vertically — same bar as building upgrade (Story 4).

---

# 4. Development Process (unchanged)

Every story follows the proven cycle:

```
User Story → Backend → Frontend → Game Loop → Realtime → E2E → Release Gate
```

| Artifact | Required |
|----------|----------|
| User Story + DoD | Yes |
| Backend (command + loop) | Yes |
| Frontend (store + UI) | Yes |
| Domain event + WS subscriber | If state changes |
| Smoke / resilience extension | Yes |
| Metrics observation | If load changes |
| Documentation | Yes |

No story ships without passing the release gate contract.

---

# 5. What v0.2 Is Not

- Not a rewrite of the game loop
- Not new queue tables per resource type
- Not infrastructure for infrastructure's sake
- Not “more numbers on dashboard” without decisions

---

# 6. Success Criteria for v0.2

Player can:

1. See production change over time (Game Loop, not HTTP polling).
2. Hit a storage or production bottleneck and react.
3. Choose between competing upgrades based on economy.
4. Stay synced via WebSocket — no F5 for resource changes.

Release: extend `smoke:e2e` + `release:gate` for economy scenarios.
